## [4] 助手 (jsonl 行 4)

**会有一定影响，但通常不会完全不能用。** Cursor 把用量分成两个独立池子，你看到的 **Total** 不是第三个额度。

## 三个数字分别是什么

| 指标 | 含义 |
|------|------|
| **Auto + Composer** | 专用于 **Auto 模式**、**Composer 1.5**、以及走这些模型的 Agent 操作。这个池子额度通常更大、更划算。 |
| **API** | 用于**手动选择**的前沿模型（如 Claude Sonnet、GPT、Opus 等），按 API 价格计费。Pro 约含 $20/月，更高档位更多。 |
| **Total** | **不是独立额度**，而是两个池子按套餐权重算出的**综合消耗百分比**，所以可能出现「Auto+Composer 用完了，但 Total 还没满」的情况。 |

## Auto + Composer 用完后会怎样

1. **Auto / Composer 1.5 的「免费额度」用完了**  
   继续用 Auto 或 Composer，会开始**消耗 API 池子**（按 API 价格计），而不是完全停用。

2. **API 还有余量时，你仍然可以正常用 Cursor**  
   - 继续用 **Auto**（会扣 API 额度）  
   - 或改用手动选的模型（直接走 API 池子）  
   两个池子是独立的，API 没用完就可以继续用。

3. **两个池子都到 100% 时**  
   通常会提示：升级套餐、开启 **On Demand**（按需付费），或等到下个账单周期重置。

## 对你当前使用的影响

你现在用的是 **Auto 模式**（本对话就是）。Auto + Composer 用完后：

- **还能继续用**，但后续请求会走 **API 额度**，消耗会比之前快  
- 若 API 也用完，才会真正受限或需要额外付费

## 建议

1. 在 **Settings → Usage** 或 [cursor.com/dashboard?tab=usage](https://cursor.com/dashboard?tab=usage) 看两个池子各自还剩多少  
2. 想省 API：可改用手动选 **Composer 1.5**（若界面仍显示有额度）或更便宜的模型  
3. 若经常用 Auto，可考虑开启 On Demand 或升级套餐，避免月中断档

**简单说：Auto + Composer 用完 ≠ Cursor 不能用；只要 API 还有余量，就能继续，只是 Auto/Composer 会开始扣 API 额度。** 只有 API 也耗尽时，才会明显受限。

## [7] 助手 (jsonl 行 7)

**是的，大多数情况下模型仍然是 Composer（或 Auto 路由到的 Composer），变的主要是扣费池子，不是模型本身。**

## 核心逻辑

Cursor 的两个池子是**计费来源**，不是两套完全不同的产品：

| 你选的模式 | Auto + Composer 有额度时 | Auto + Composer 用完后 |
|-----------|-------------------------|------------------------|
| **Auto** | 走 Auto + Composer 池 | 仍走 Auto，但**改扣 API 池** |
| **手动选 Composer 1.5 / 2 / 2.5** | 走 Auto + Composer 池 | **仍是 Composer**，但**改扣 API 池** |
| **手动选 Sonnet / GPT 等** | 从一开始就走 API 池 | 继续走 API 池 |

官方社区有用户实测确认：Auto 池用完后，继续用 Composer 2 / Composer 2 Fast 是**无缝的**，只是消耗从 Auto 池转到 API 池，一般不会有弹窗或额外收费（前提是未开启 On Demand 按需付费）。

## 所以对你问题的直接回答

> Auto + Composer 用完，用的是 API 额度，但模型仍然是 Composer 吗？

**对。** 只要你继续选的是 **Auto** 或 **Composer** 系列，底层通常还是 Composer；只是这笔用量不再算在「Auto + Composer 赠送额度」里，而是算进你的 **API 月度额度**（例如 Pro 约 $20）。

## 需要注意的例外

1. **API 也用完时**（且未开 On Demand）：请求会被拦截，或提示升级/开启按需付费，这时才真正「不能用」。
2. **你手动选了 GPT / Sonnet 等前沿模型**：本来就走 API 池，和 Auto + Composer 是否用完无关。
3. **API 用完、你还坚持选前沿模型时**：系统可能自动 fallback 到 Composer，并弹出类似 “Switched to Composer after reaching API limit” 的提示——这是**反过来的情况**（API 不够，退回 Composer），不是你现在问的场景。

## 简单记忆

- **模型**：你选什么，大概率还是什么（Auto → 通常还是 Composer 系）
- **额度**：Auto 池用完 → 同一模型改扣 API 池
- **真正断档**：两个池子都 100%，且没开 On Demand

你可以在 [cursor.com/dashboard?tab=usage](https://cursor.com/dashboard?tab=usage) 看每次请求实际用的是哪个模型、扣的是哪个池子。

## 日志里有什么

`Power.log` 会为每个英雄技能创建 `CARDTYPE=HERO_POWER` 实体，解析后落在 `GameState.entities` 里，可读字段包括：

| 字段 | 含义 |
|------|------|
| `card_id` | 如 `HERO_08bp`（法师火焰冲击）、`HERO_04bp`（圣骑征召） |
| `cost` | 当前费用（双职业常见为 2） |
| `zone` | `PLAY` = 当前可用；`SETASIDE` = 被替换掉的主职业技能 |
| `EXHAUSTED` | 本回合是否已用（`1` = 已用） |
| `LINKED_ENTITY` | 双职业时指向主职业技能实体 |
| `CLASS` | 技能所属职业 |

我用你本地的双职业对局日志 `game_05.log` 验证过，结构如下：

```
玩家1（DH 英雄）:
  SETASIDE: HERO_10akhp  (恶魔之爪，主职业)
  PLAY:     HERO_04bp    (征召，副职业圣骑) ← 当前可用

玩家2（DK 英雄）:
  SETASIDE: HERO_11bp    (食尸鬼冲锋，主职业)
  PLAY:     HERO_08bp    (火焰冲击，副职业法师) ← 当前可用
```

双职业开局会触发 `FORCE_DISPLAY_CLASS_UI`：主职业技能进 `SETASIDE`，副职业技能进 `PLAY` 并带上 `LINKED_ENTITY`。

## 「我能用的技能」怎么判

每回合实际能点的是：

> 该玩家 `zone == "PLAY"` 且 `cardtype == "HERO_POWER"` 的那一个实体  
> 且 `EXHAUSTED != 1`、法力够 `cost`

不是看英雄卡上的 `HERO_POWER` 标签（那是主职业默认技能，双职业时往往已在 `SETASIDE`）。

`card_database.HERO_POWER_DAMAGE` 也只有 4 条旧 ID（如 `CS2_034`），没有 `HERO_08bp` 等竞技场常用技能。

## 若要做英雄技能模块，建议第一步

在 `GameState` 加两个方法即可把双职业信息读全：

```python
def get_active_hero_power(self, player_id: int) -> Optional[Entity]:
    """当前可用英雄技能（双职业 = PLAY 区的副职业技能）"""
    for e in self.entities.values():
        if e.cardtype == "HERO_POWER" and e.zone == "PLAY":
            if self.is_entity_controlled_by(e, player_id):
                return e
    return None

def get_hero_power_pair(self, player_id: int) -> tuple[Optional[Entity], Optional[Entity]]:
    """(当前可用, 主职业被替换的技能)"""
    active = primary = None
    for e in self.entities.values():
        if e.cardtype != "HERO_POWER":
            continue
        if not self.is_entity_controlled_by(e, player_id):
            continue
        if e.zone == "PLAY":
            active = e
        elif e.zone == "SETASIDE" and not e.tags.get("LINKED_ENTITY"):
            primary = e  # 或被 LINKED_ENTITY 指向的那个
    return active, primary
```

然后按 `card_id` 建 `hero_power_p0.py`（直伤、召唤、加攻等），类似武器/战吼模块。

## 实测结论（`game_06.log`）

年兽 `EDR_421` 每次**完成一次攻击**后，会写入：

```
TAG_CHANGE ... tag=TAG_SCRIPT_DATA_NUM_1 value=N
```

同时 `tag=4741` 会与 `TAG_SCRIPT_DATA_NUM_1` 保持相同数值。

| 状态 | `TAG_SCRIPT_DATA_NUM_1` | 亡语伤害 |
|------|-------------------------|----------|
| 刚上场，未攻击 | 1（或仅有 `4741=1`） | **1** 点全体敌人 |
| 攻击 1 次后 | 2 | **2** 点 |
| 攻击 2 次后（风怒当回合） | 3 | **3** 点 |
| 下回合再攻击 1 次 | 4 | **4** 点 |

`game_03.log` 里年兽上场后没来得及攻击就被消灭，亡语打出 **1** 伤，与 `4741=1`、未更新 `TAG_SCRIPT_DATA_NUM_1` 一致。

```python
omen = gs.get_entity(entity_id)  # 或遍历 board
```

## 正确模拟方式

**抹除存在** `TIME_433` 现在这样算：

1. **每次尝试**：从**同一初始场面**出发，**随机**沉默 + 消灭 1 个敌方随从  
2. **回溯**：共模拟 **2 次**（第一次 + 回溯重来一次）  
3. **取舍**：比较两次结果的场攻，**保留更好的一次**  
4. 仍标记 `uses_random=True`（本质仍是随机，只是多一次机会）

## 实现要点

```python
# 从初始场面克隆 → 随机消灭 → 计分；做 2 次，取 max
_apply_rewind_random_silence_destroy(..., attempts=2)
```

- 无 `rng` 时：用 `Random(0)`、`Random(1)` 两次独立抽样  
- 有 `rng`（MC）时：每次回溯用**独立子随机源**，避免共用同一个 `rng` 导致两次抽到同一目标  

## 示例
